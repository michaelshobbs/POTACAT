# The Big Beautiful Hamvention Build

Headline: ragchew logger upgrades from the Discord thread (collapsible past QSOs, beam heading, smaller window, one-click DX cluster spot), Ctrl/Alt-modified F-key hotkeys are no longer eaten by the F1 help dialog, ECHOCAT now shows your real CW macros instead of the defaults, FT8 stops decoding far less often (and recovers itself when it does), and ECHOCAT finally has a CW Stop button.

If you're a Flex user and CW macros from the desktop have ever silently done nothing while a WinKeyer was plugged in — there's a routing fix in here for you too.

## Ragchew logger pop-out — the W9TEF + KM4CFT thread

Five items from the Discord ragchew thread shipped here:

- **Past QSOs is collapsible.** Click the "Past QSOs (N)" header to fold the panel away when you want the form to take less space, click again to bring it back. The state persists per-window in localStorage, so once you set it, it stays.
- **Beam heading next to the identity line.** When the entered callsign has a grid in QRZ and your station grid is set, the identity line now reads "Bob McDonnell · US-VA · Beam 087°". KM4CFT asked for this so he doesn't have to bounce out to QRZ every QSO; the bearing is computed locally from grids using the same great-circle math the map uses.
- **Smaller responsive window.** The popout's min size dropped from 460×560 to 360×420, and the form rows now flex-wrap so a narrow column doesn't clip fields. You can park the popout in a tight corner of your screen and the date/time + frequency/mode rows just stack instead of getting cut off.
- **Past QSOs date column won't clip.** The table is wrapped in a horizontal scroll container, so even at the new narrow widths the date+time stays readable instead of being chopped off.
- **Auto-reopens on launch.** If the logger window was open when you quit POTACAT, it'll come back when you relaunch. Same pattern as the VFO popout and the QSO Logbook popout — no new setting to manage.

And one bonus item from the same thread: **Tom's "Spot on DX Cluster" idea.** When the **DX** chip is selected and you have at least one cluster node connected, the popout now shows a "Spot this DX on the cluster" toggle. Save the QSO and POTACAT posts a `DX <freq> <call> <comment>` line to every connected node, using the new dxRespotTemplate setting (defaults to `Heard in {QTH} 73s {mycallsign} via POTACAT`). Same pattern as the existing POTA/WWFF/LOTA re-spot toggles, so the muscle memory carries over.

## Ctrl+F1 / Alt+F1 hotkeys actually fire now (N4BJH)

If you'd bound `Ctrl+F1` (or any Ctrl/Alt+F-key) to tune a frequency via the user hotkey editor, v1.5.18 silently swallowed your binding and showed the F1 help dialog instead. The renderer's F-key handlers weren't checking for modifier keys, so the bare-F1 path always won. Fixed for F1, F2, F4, F5, and F11 — modifier-combos now fall through to the user hotkey table where they were always meant to land. Shift+F4's intentional "mega celebration" branch still works.

## ECHOCAT — your CW macros, not the defaults (KK4DF)

Walt reported that after upgrading to v1.5.18, the CW Macros pane in ECHOCAT showed the factory defaults instead of his custom macros. The cause was a settings-key split that built up gradually: desktop POTACAT writes macros to `settings.cwMacros` (the Settings dialog editor), while ECHOCAT's phone-pushed edits land in `settings.remoteCwMacros`. The desktop only pushed `remoteCwMacros` to the phone, so anyone who'd only ever edited macros on the desktop saw defaults on the phone.

Two changes:

- ECHOCAT now uses the desktop's `cwMacros` as a fallback when `remoteCwMacros` is empty. Phone-pushed macros still win when set, so anyone whose phone macros diverged from the desktop keeps both.
- Saving CW macros in desktop Settings → CW now immediately pushes to any connected phone. Previously the push only fired on unrelated triggers (rotor toggle, custom CAT buttons), so the phone could stay stale until the next reconnect.

No migration step — your desktop macros just start showing up on the phone after the upgrade.

## FT8 — worker watchdog catches silent hangs (OneLD)

Luk reported FT8 quietly stopping mid-session: waterfall still drawing, audio still flowing, but no decodes appearing in the table. Restart fixes it. The existing safety nets in JTCAT didn't catch this case because they assumed the worker would either crash (exit handler fires) or stay responsive enough to send back zero-decode results (the WASM stall detector triggers). A worker thread that hangs *without* crashing — the native FT8 addon getting stuck, a GC pause that never resolves, a queue saturation that backs up — slips between both nets.

v1.5.19 adds a heartbeat watchdog: every message the worker sends back (ready, decode-result, encode-result, even errors) refreshes a `_lastWorkerResponseMs` timestamp. At each cycle boundary, if more than 2.5 cycle lengths have passed without a response (~38s for FT8, ~19s for FT4), POTACAT logs `FT8 worker unresponsive for Ns — respawning (watchdog fire #N)` to the verbose CAT pane and respawns the worker. The fresh worker re-seeds the timestamp on spawn so the watchdog grants the new instance a full grace window before re-evaluating.

If you've been hitting silent FT8 stalls, you'll see them auto-recover now, and the log line gives us real signal (how often, after how long) for tracking down the root cause. The 2.5-cycle threshold has 1.5 cycles of slack baked in to absorb GC pauses and slow disks before declaring the worker dead, so this shouldn't false-positive during a normal slow decode.

## ECHOCAT — CW Stop button (AA6C)

Josh asked for a way to cancel a CW macro that's already in flight: tap the wrong button and you used to have to wait it out. The desktop's CW widget had an ESC button already; ECHOCAT didn't.

New red **Stop** button next to **Send** in the ECHOCAT CW pane. Tap it and POTACAT aborts whatever's in flight across every CW backend it knows about: WinKeyer buffer flush, SmartSDR `cwx clear`, Python pyserial helper SIGTERM, node-serialport DTR-timer clear with a forced key-up, and CAT-side `stopCwText` (CI-V 0x17 0xFF, plus a new path that drops PTT for Yaesu/Kenwood/rigctld).

The desktop ESC button benefits too — it was missing the pyserial / DTR-timer / CAT-side cancels, so users on those rigs effectively had a non-functional cancel. All three desktop paths (main window ESC, VFO popout cancel, phone Stop button) now route through one shared `cancelAllCwSends()` function that hits every backend, so they can't drift apart.

## CW dispatch priority — Flex users with paddle-only WinKeyer (K3SBP)

Caught this one in dev testing the Hamvention week. If you have a WinKeyer connected on a COM port for paddle work and run a Flex 6000-series radio, POTACAT was sending your CW macros to the WinKeyer's serial port — which is correct behavior if the WinKeyer is physically wired to the Flex's KEY jack, but silent failure if it's a USB-only setup for paddle keying. Your "QRZ?" macro looked like it transmitted (the log said `[CW] Text: QRZ?`) but the radio never keyed.

New priority: when the active rig is a Flex and SmartSDR's CW client-bind succeeded (`SmartSDR] cw auth: method=bind ok=true` in your log), macros now go through SmartSDR's `cwx` path. WinKeyer stays the default for non-Flex rigs (CI-V, Yaesu, Kenwood, rigctld) where CAT-side CW has firmware quirks WinKeyer reliably sidesteps. Flex users who *did* wire WinKeyer to the Flex's KEY jack hear no difference — both paths key the same radio, just over different transports.

`WinKeyer.sendText` also logs the routed characters and WPM now (`[WinKeyer] sendText("QRZ?") @ 25 WPM`), so a silently-eaten send is no longer indistinguishable from a working one. Next time someone hits this on a different rig combo, the routing will be obvious in the verbose log.
