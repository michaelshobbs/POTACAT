# The VFO stops flickering backward when you spin it

**Headline:** Spin the VFO knob — on the desktop popout, in the web ECHOCAT, or in the native iOS app — and the dial **stays where you put it.** Two reporters (W8IJW on an IC-7300, K6RBJ across phone + PC) hit the same race: the rig accepted each tune but the next CAT poll returned the freq from ~one wheel-tick ago and the display snapped backward mid-spin. Fixed at both surfaces.

## What was happening

The wheel handler did the right thing — it updated the display optimistically and shipped the tune to the rig. But between the tune going out and the rig settling, POTACAT's `FA;` poll (or the equivalent CI-V read) completed and returned the rig's freq from one tick ago. The display would then snap from 14.281 back to 14.280, then forward again on the next user click, then back again on the next poll. Visibly broken; functionally the rig was actually moving where the user pointed it — the dial display was just lying.

## How it's fixed

The same defense the popout already had for `mode` (a brief suppress window after a user-initiated change), now applied to `freq`:

- **Desktop VFO popout** — every user-initiated tune (wheel / drag / keypad / band button / profile recall) now bumps a `_freqSuppressUntil` window. Incoming `radio-state` updates with stale freq are ignored for 1.2 s. After that, the polled value — by now reflecting the rig's actually-settled freq — flows normally. External knob spins on the rig itself still propagate within ~1.2 s; they don't reset the window.

- **Phone + web ECHOCAT** — same idea, applied at the server so both clients are covered without touching either. When the connected client sends `'tune'`, the server replaces `freq` in subsequent status broadcasts with the client's own tune target for the next 1.2 s. Mode, S-meter, SWR, ALC, audio health all still flow live — only `freq` gets rewritten.

That's the whole release.
