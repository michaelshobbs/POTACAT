# VFO snap-back: wait for the rig to actually arrive

**Headline:** v1.7.2 hid the snap-back during a wheel spin, but the moment the suppression timer expired the display would still snap to whatever the last CAT poll happened to read — and if the rig hadn't physically caught up to the user's target by then, you'd see the dial jump backward anyway. v1.7.3 holds the display at the user's target **until the rig confirms it arrived**, on both the desktop popout and the phone/web ECHOCAT.

## The actual fix

The old fix used a fixed 1.2-second window: ignore polled freq for 1.2 s after a tune, then accept whatever. That assumed the rig would always catch up in time. For users wheeling fast or running CI-V at slower baud rates, it didn't.

v1.7.3 is match-based:

- **Hold the optimistic display** (popout) — or **echo the client's tune target** (ECHOCAT server) — until the polled freq lands within 25 Hz of what the user asked for. That's the rig confirming "I'm there." Release happens on the very next matching poll, so a fast-responding rig sees no perceptible delay.
- **3-second hard timeout** as a safety net — if the rig refuses to confirm (something genuinely wrong), the display releases anyway so a broken rig can't deadlock the dial.
- External knob spins on the rig itself still propagate immediately — they only happen when no target is armed, so the match check is bypassed.

Same change on both surfaces, so all three VFOs — desktop popout, web ECHOCAT, native iOS app — benefit from one server-side change plus the local popout change.

## CI-V diagnostic logging for Icom users

If you're on an Icom (IC-7300 / IC-705 / IC-7610 / IC-9700 / IC-705) and you're still seeing the dial snap to a previous freq even with v1.7.3, the verbose log now records **every** freq write at the CI-V layer (lines like `setFrequency 14275000 Hz`). The first such line that appears after you stop wheeling identifies what's commanding the rig. Grab the log when you hit the bug and share it on Discord — that one line will pin the root cause down.

This is purely diagnostic — no behavior changes for Icom users.
