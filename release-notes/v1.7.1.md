# Flex Direct, now with AetherSDR (and a rig switch that actually rigs)

**Headline:** v1.7.0 made POTACAT a Flex GUI client when nothing else was running, but missed the case the most users actually live in — **AetherSDR (or SmartSDR-Win) already up, POTACAT alongside.** And on the desktop, switching to a Flex rig from the editor didn't reconnect the FlexLib API client at all, so the audio came up but tunes returned "no radio connected." Both fixed. Plus: voice macros import audio files and preview locally, the mode filter is strict again when you ask it to be, the banner logger accepts kHz, and the space-weather pills survive hamqsl going down.

## Flex: bound-mode is a first-class path now

Casey running an 8600 with AetherSDR open hit two bugs ganging up. The desktop rig editor's "Apply" never reconnected the SmartSDR API client (only the phone's switch-rig handler had done that); and even after that, *bound mode* — POTACAT bound to AetherSDR's GUI client rather than being its own — never tracked an active slice, so every code path keyed on slice readiness was dead. Tunes were ignored, frequency never displayed, the CAT pill stayed dim.

v1.7.1 makes bound mode a first-class path:

- **Desktop rig switch** now starts the FlexLib API client the same way the phone's switch-rig always has. `connect-cat` triggers `connectSmartSdr()`, and the save-settings path picks it up too as belt-and-suspenders.
- **Bound mode tracks the host's active slice** — both AetherSDR and SmartSDR-Win flag exactly one slice per panadapter as `active=1`, and POTACAT follows it. Change the active slice in AetherSDR and POTACAT's tune target moves with it.
- **The tune branch and the CAT pill** key off a new "can drive a slice" check that's true for both self-host (Flex Direct) and bound modes. CAT pill lights, frequency displays, tunes land on the radio.
- The CAT log distinguishes `tune via Flex Direct` (self-host) from `tune via Flex API (bound)` so you can tell at a glance which path your commands took.

## Voice macros — load your Audacity takes, preview before you send

Two operator-driven additions to **Settings → Voice Macros**:

- **Import** — load a pre-recorded audio file (WAV, MP3, OGG, Opus, FLAC, M4A) into a macro slot. The file is test-decoded before saving, so a bad pick fails fast with a clear message instead of silently breaking the slot. Trim and EQ your CQ macro in Audacity exactly the way you'd build a SmartSDR / AetherSDR DVK take, then drop it into POTACAT.
- **Preview** — play any filled slot through your computer speakers without PTT'ing the rig. Click again to stop. Audio routes through your default output, not the rig device.

State for the local preview is isolated from the TX-bound playback path so an accidental Preview can't collide with a transmission in progress.

## Hunter Mode — strict CW filter is strict again

Filter the spot table to "CW only" and you'd see mode-less POTA spots leak through. They aren't CW — mode-less means *we don't know what this is* — so for an operator hunting a specific mode they were noise. v1.7.1 hides empty-mode spots under any specific mode filter. Want to see them anyway? Use the "All" mode filter — that's what it's for.

## Banner logger accepts kHz or MHz

The bottom-bar quick-logger's frequency field used to require MHz ("14.275"). The rest of POTACAT works in kHz everywhere — POTA spots, the CAT log, the status bar — so typing kHz here was the natural reflex, and a "7200" entry intended for 40m went out as 7200 MHz, fell outside every ham band, and got rejected by Wavelog with "no band." v1.7.1 accepts either: integers ≥ 1000 are read as kHz, decimal values pin to MHz, the only ambiguous VHF/UHF integers (50, 144, 432, 1296) only validate one way. Bad input flashes the field red with a clear message instead of forwarding garbage.

## Space-weather pills survive a hamqsl outage

hamqsl.com occasionally serves an HTML error page when it's overloaded — the old code bailed entirely the moment any of SFI / A / K didn't parse, and the three pills stayed em-dashed indefinitely with no log line explaining why. v1.7.1:

- Takes whatever hamqsl gave us instead of all-or-nothing.
- Falls back to **NOAA SWPC** for the rest — 10.7 cm flux for SFI, the existing 24-hour Kp history for the K pill, and an Ap derived from the same 8 Kp samples (standard Kp→ap table) for the A pill.
- Carries over previously-cached softer fields (X-ray, MUF, aurora, etc.) instead of blanking them.
- Logs the upstream failure once per fetch so a stuck pill is no longer mysterious.

## Other improvements

- **SSTV decoder** — every decode-end now surfaces `mode`, sync %, and the column-spread metric (on both passing images and rejections). When a noisy or wrong-color decode comes through, the log has the numbers we'd otherwise have to ask for in Discord.
- **ECHOCAT pairing** — defensive `lastSeen` stamp covers legacy-token + pre-existing pair entries.

**macOS users on Apple Silicon:** the DMG is ad-hoc signed (no paid Developer ID yet). If you see "POTACAT is damaged and can't be opened," run this once from Terminal and relaunch:

```
sudo xattr -dr com.apple.quarantine /Applications/POTACAT.app
```

(In-app auto-update is disabled on macOS — grab the new DMG from the GitHub release directly.)
